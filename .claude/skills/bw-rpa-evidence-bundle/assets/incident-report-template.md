## 症状（What happened）

### 発生した現象

（簡潔に1-3文で記述）

例:
- 楽楽精算への経費申請登録処理が30秒でタイムアウトし、失敗しました。
- 全10件中、5件目で処理が停止しました。

### エラーメッセージ

```
（エラーメッセージ全文をここに貼り付け）
```

### 影響

- 処理件数: 成功 {n}件 / 失敗 {m}件 / 未処理 {k}件
- データ整合性: 整合 / 不整合（詳細: ...）
- 業務への影響: なし / あり（詳細: ...）

---

## 原因（Why it happened）

### 根本原因

（現時点で判明している原因を記述）

例:
- UI変更: 楽楽精算のログインボタンのセレクタが `button[type="submit"]` から `#login-btn` に変更されていた
- ネットワーク: 社内ProxyのSSL証明書エラー
- 権限: 実行ユーザーのアカウントが無効化されていた
- タイムアウト: 通常5秒で完了する処理が30秒経過しても応答なし

### 一次原因

（根本原因の前段階で発生した事象）

例:
- セレクタが見つからず、Playwright が TimeoutError を投げた

## 対応（What was done）

### 暫定対応（Workaround）

（即座に実施した対応。恒久対策ではないもの）

例:
- セレクタを手動で修正し、再実行
- タイムアウト時間を60秒に延長
- 手動で残り5件を処理

**実施日時**: YYYY-MM-DD HH:MM
**実施者**: {氏名}
**結果**: 成功 / 一部成功 / 失敗

### 恒久対策（Permanent fix）

（根本原因を解決する対策）

例:
- セレクタをより安定したもの（`data-testid` 属性）に変更
- Proxy設定を修正し、SSL証明書エラーを解消
- 実行ユーザーのアカウント状態を定期監視
- タイムアウト設定を業務特性に合わせて調整（標準30秒 → 60秒）

**実施予定日**: YYYY-MM-DD
**担当者**: {氏名}

## 再発防止（Prevention）

### 検出方法の改善

（今後、同様の問題を早期に検出する方法）

例:
- セレクタ変更検知: 定期的にUI構造をスナップショットし、差分を検知
- ヘルスチェック: 毎朝、主要ページへのアクセステストを実行
- アカウント監視: 実行ユーザーのアカウント状態を毎日確認

### 設計の改善

（今後、同様の問題が発生しにくくする設計変更）

例:
- セレクタ優先順位: `data-testid` > `role` > `id` > `class` の順に試行
- リトライロジック: ネットワークエラー時は最大3回リトライ
- タイムアウト設定: 業務ごとに適切な値を設定（設定ファイル化）

### ドキュメント更新

（更新すべきドキュメント）

- [ ] README.md: 新しいセレクタ戦略を追記
- [ ] TROUBLESHOOTING.md: 本障害を追加
- [ ] CONFIG.md: タイムアウト設定を明記
