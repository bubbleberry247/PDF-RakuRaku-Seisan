# Operating Model（運用体制定義）v0.3

**最終更新**: 2026-01-18
**ステータス**: 確定（OM3）

---

## 1. 体制タイプ

| タイプ | 説明 | 選択 |
|--------|------|------|
| OM1 | 自社（開発者）が運用者。manual_queue含めて自社で処理 | - |
| OM2 | 自社は自動化運用のみ、顧客がmanual_queue処理（最終責任は顧客） | - |
| **OM3** | **顧客が運用者、自社は保守/改善のみ（監視と障害対応の範囲を限定）** | **✓** |

**確定**: OM3（顧客運用、開発者は保守/改善のみ）

---

## 2. 役割定義（Role）

| 役割 | 責務 | 割当（OM3） |
|------|------|------------|
| **Owner** | 業務の最終責任者（品質/期限/監査） | **顧客** |
| **Operator** | 日次運用の実行者（自動実行、一次対応） | **顧客** |
| **Reviewer** | manual_queueの目視確認・補完 | **顧客** |
| **Approver** | 最終確定（必要なら） | **顧客** |
| **Maintainer** | 改善/修正/リリース担当 | **開発者** |

### 2.1 責任分界

| 領域 | 顧客 | 開発者 |
|------|------|--------|
| 日次運用（実行/監視） | ○ | × |
| manual_queue処理 | ○ | × |
| 障害一次対応（切り分け） | ○ | △（相談対応） |
| 障害二次対応（修正） | × | ○ |
| ログ/データ保管 | ○ | △（技術支援） |
| 改善/機能追加 | △（要望出し） | ○ |
| リリース作業 | △（受入確認） | ○ |

---

## 3. RACI（主要タスク）

| タスク | Owner(顧客) | Operator(顧客) | Reviewer(顧客) | Maintainer(開発者) |
|--------|-------------|----------------|----------------|-------------------|
| 日次実行（自動） | A | R | - | C |
| manual_queue処理 | A | - | R | C |
| 障害一次対応（切り分け） | A | R | - | C |
| 障害二次対応（修正） | I | - | - | R |
| ログ/データ保管 | A | R | - | C |
| 監査ログ確認 | A | - | R | C |
| 改善要望 | R | - | - | A |
| 改善/リリース | A | - | - | R |

**凡例**: R=実行責任, A=説明責任, C=相談, I=報告先

## 4. SLA（Service Level Agreement）

### 4.1 シナリオ別処理頻度

| シナリオ | 頻度 | タイミング | 備考 |
|---------|------|-----------|------|
| S37（レコル公休・有休登録） | **月1回** | 月末締め→月初提出 | 必要なら毎日も可能だが月1回で十分 |
| S44（楽楽精算PDF処理） | 随時 | 申請発生時 | - |
| S47（出退勤確認） | TBD | TBD | - |

**根拠（S37）**: 2026-01-16 瀬戸さんヒアリング

### 4.2 日次処理（顧客責任）

| 指標 | 目標値 | 備考 |
|------|--------|------|
| 自動処理実行時間 | 顧客が決定 | スケジューラ設定は顧客管理 |
| 自動処理完了目標 | 顧客が決定 | - |

### 4.3 Manual Queue（顧客責任）

| 指標 | 目標値 | 備考 |
|------|--------|------|
| 初動（確認開始） | 顧客が決定 | 推奨: 当日中 |
| 完了（登録確定） | 顧客が決定 | 推奨: 翌営業日 |
| 滞留アラート閾値 | 顧客が決定 | 推奨: 5件 |

### 4.4 保守対応（開発者責任）

| 指標 | 目標値 | 備考 |
|------|--------|------|
| 問い合わせ初動 | 1営業日以内 | 調査開始の連絡 |
| 障害修正（緊急） | 3営業日以内 | 業務停止レベル |
| 障害修正（通常） | 5営業日以内 | 業務継続可能 |
| 改善/機能追加 | 要相談 | 規模による |

### 4.4 エスカレーション

| 状況 | 顧客アクション | 開発者アクション |
|------|--------------|-----------------|
| 自動処理エラー | 切り分け、ログ収集 | 調査依頼を受けて対応 |
| manual_queue滞留 | 社内エスカレーション | - |
| システム停止 | 開発者へ連絡 | 即時調査開始 |
| パターン追加が必要 | 要望連絡 | 改善対応 |

## 5. 権限・機密

### 5.1 認証情報管理

| 対象 | 管理者 | 備考 |
|------|--------|------|
| 楽楽精算ログイン | **顧客** | 開発者には共有しない |
| Adobe PDF Services API | 開発者 | APIキーは開発者管理 |
| システムログイン（PROD） | **顧客** | 開発者は必要時のみアクセス |

### 5.2 アクセス権限

| 環境 | 顧客 | 開発者 |
|------|------|--------|
| PROD（本番） | フルアクセス | **原則アクセスなし**（障害時のみ依頼ベース） |
| LOCAL（検証） | - | フルアクセス |

### 5.3 機微情報の取り扱い

- ログへの機密情報出力: `docs/logging_policy.md` 準拠
- PDFファイル保管: **顧客管理**
- 処理済みファイルの削除: **顧客ポリシーに従う**
- 障害解析時のログ共有: **顧客が判断**（開発者からは要求しない）
