- `名古屋ターミナルビル_20250125_54000.pdf`

**~~結論~~**: ~~PDF OCRは不要。添付ファイル名からの抽出が最も信頼性が高い。~~

### 2026-02-18 [G8] 全量ファイル名規約テスト結果（Supersedes上記結論）

**対象**: NoFlag(0) + 件名フィルタ通過の全PDF添付ファイル149件

| パターン | 件数 | 割合 | 説明 |
|---------|------|------|------|
| A: `業者名_日付_金額_工事名.pdf` | 66 | 44.3% | 完全形（ルーティング可能） |
| B: `業者名_日付_金額.pdf` | 33 | 22.1% | 工事名なし（リネームのみ可能） |
| D: 非準拠 | 50 | 33.6% | 規約外（フォールバック必要） |
| ZIP | 9 | — | 別途処理 |

**非準拠の内訳**:
- 日付フォーマット違い（30%）: `2026.1.25`, `260125`, `2026_0125`
- 区切り文字違い（25%）: ハイフン、半角スペース
- 数値フォーマット違い（20%）: `1.650,000`（ドイツ式）、`￥150,000`
- スキャナー自動命名等（25%）: `SKM_C251i...`, タイムスタンプ、業者名のみ

**修正結論**: filename-first(66%) > PDF OCR(30%) だが、**緩いパーサー + PDFフォールバックの両方が必須**。
- **ルーティング用**（工事名）: Pattern Aの44.3%のみ。残りはPDFフォールバックか「その他」
- **リネーム用**（vendor/date/amount）: 66.4%がファイル名から取得可能

**方針変更**: 「filename-first, PDF fallback」は維持するが、ファイル名パーサーは複数フォーマット対応の緩い実装が必要。

## 2026-02-17 [DESIGN] 3文字一致フォルダ振り分けロジック（Codex合意済み）

### Codex設計相談結果

**方針**: LCS（Longest Common Substring）ベースの3文字一致 + 3-gram Jaccard係数

### アルゴリズム

```
1. 添付ファイル名からキーワード抽出（NFKC正規化）
2. 保存先ルート内の既存フォルダ名一覧を取得
3. 各フォルダ名との類似度スコア計算:
   - LCS長（最長共通部分文字列）≥ 3 が最低条件
   - 3-gram Jaccard係数で精密スコアリング
4. ルーティング判定:
   - 最高スコアのフォルダが閾値以上 → そのフォルダに保存
   - 僅差（上位2つの差が小さい） → 「その他」に保存（ポカヨケ）
   - 閾値未満 → 「その他」に保存
```

### 設定項目（tool_config拡張）

```json
{
  "routing": {
    "existing_folders_root": "\\\\192.168.1.251\\TIC-mainSV\\経理\\請求書【現場】",
    "match_min_lcs": 3,
    "match_threshold": 0.3,
    "tie_margin": 0.05,
    "unknown_folder": "その他",
    "source_priority": ["filename", "pdf_text"]
  }
}
```

### アーキテクチャ

```
Filename extraction (primary)
  → NFKC normalize
  → LCS + 3-gram scoring vs existing folders
  → Route to best match / "その他"

PDF text extraction (fallback, only if filename insufficient)
  → PyMuPDF → regex patterns
  → Same scoring pipeline
```

### ポカヨケ設計

| リスク | 対策 |
|--------|------|
| 複数フォルダが同スコア | tie_margin内なら「その他」に退避 |
| 新しい工事名（フォルダなし） | LCS < 3 なら「その他」に退避 |
| ファイル名が無意味（hash等） | PDFテキストにフォールバック、それでも不可なら「その他」 |
| 「その他」が溜まる | 月次レポートで未振り分け件数を通知（アンドン） |

### レビュー概要

### Critical指摘（2件）

**C1: Dry-runの偽りの安心感**
- Dry-runではvendor/date/amountを`None`で記録（py:2513-2515）、実際の抽出・リネーム本体を通さない
- 「0エラー」は復号・ZIP展開・OCR/PDF解析・リネーム精度・ファイル移動・印刷を一切検証していない
- **対策**: `--execute --max-messages 10` で代表サンプルの実機通しが必須

**C2: ドキュメント間の矛盾（3資料が不整合）**
- decisions.md: 「ファイル名優先」と結論（line 86）
- runbook: PDF-SSOTのまま（s12_13...md:93）
- 実装コード: PDF本文抽出ベースのまま（py:929, 1935, 1945）。添付ファイル名パース処理は未実装
- **対策**: 3資料を「filename-first + PDF fallback」で統一

### High指摘（6件）

**H1: 既読+NoFlagの44通がスキップされるリスク**
- デフォルト`unread_only=True`（py:248）。テストconfigでは`false`だが本番configで忘れると44通漏れ
- **対策**: 本番configで`unread_only: false`を明示的に設定、preflight checkに追加

**H2: FlagStatus読取失敗時に`0`（未処理）扱い**
- COM例外時に`flag_status=0`にフォールバック（py:2388-2390）
- COM不安定時に処理済みメールを再処理する誤動作リスク
- **対策**: 例外時は`flag_status=-1`（スキップ）として、ログに警告出力

**H3: 部分成功時の冪等性が弱い**
- 1通に複数添付 → 一部成功・一部失敗 → フラグ未変更 → 再実行で重複保存（`__dup`ファイル生成）
- **対策**: 添付ファイル単位のSHA-256重複チェック、または処理済みログの永続化

**H4: ファイル名優先の結論が10件サンプルに基づく**
- 163件の添付ファイルに対し10件のみ検証（6%）
- **対策**: 全163件のファイル名規約適合率を測定してから設計確定

**H5: URL付きメールのURLがreport.jsonに漏れる**
- 本文URLを収集（py:2453）→ report.jsonに格納 → メール通知に添付
- 署名付きURL/セッショントークンが外部に露出するリスク
- **対策**: report.jsonのURL部分をマスキング、またはURL-onlyタスクを別ファイルに分離

**H6: テスト不足**
- 自動テスト実質1本のみ（`test_...extract_invoice_fields.py`）
- Runbook V3-V8が未検証
- **対策**: Flag動作、URL-only停止、ZIP/パスワード、冪等性の統合テスト追加

### Medium指摘（2件）

**M1: Flagged(2)の意味が二重化**
- 調査結果: Flagged=手動作業中。しかしRPA成功時にも自動でFlagged(2)を付与
- 運用監査時に「手動中」と「RPA処理済」が区別不能
- **対策案**: Outlook Categories等の独自フィールドでRPA処理マークを追加

**M2: パスワード候補ロジックの設定値未使用**
- `decrypt_password_window_minutes`は設定読込されるが未使用（py:303, 1317）
- Stage3が「送信者無視・時刻のみ10分」で緩い
- **対策**: 設定値を実ロジックに反映、または不要なら設定項目を削除

## 2026-02-18 [DISCOVERY] OCR実装の現状分析

### 現在のカスケード（`_extract_invoice_fields_from_pdf` L944-992）

| 順序 | メソッド | 種別 | 速度 | 精度（請求書） |
|------|---------|------|------|---------------|
| 1 | pypdf | テキスト抽出 | 即時 | 30%（スキャンPDFは失敗） |
| 2 | PyMuPDF | テキスト抽出 | 即時 | 30%（同上） |
| 3 | Tesseract | 画像OCR | 5-10秒 | 中（日本語精度やや低） |
| 4 | YomiToku | 画像OCR | 60-100秒 | 中（表構造認識は良いが業者名誤読） |

### 問題点
- **pdf-ocrスキル未統合**: 既存の`PDFOCRProcessor`と独立した実装
- **EasyOCR未使用**: `_EASYOCR_AVAILABLE`チェックはあるがカスケードに組み込まれていない
- **yomitokuが最後**: 最重量のyomitokuが最終手段で、到達前にテキスト抽出で解決できないケースが多い
- **カスケード知見**: pdf-ocrスキルでは「yomitoku → EasyOCR（ページ単位）→ EasyOCR（従来）」の順序を推奨

### OCR改善の方向性（要検討）
1. **Tesseract優先化**: テキスト抽出失敗後、軽量なTesseractを先に試す（現状通り）
2. **EasyOCR追加**: Tesseract失敗時にEasyOCRを挟む（yomitokuの前に）
3. **yomitoku条件付き**: yomitokuは表構造が必要なケースのみ使用（日常の請求書には過剰）
4. **DPI最適化**: pdf-ocrスキルではDPI 3.0が最適と判明済み
