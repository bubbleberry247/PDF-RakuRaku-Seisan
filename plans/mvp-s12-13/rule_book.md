## B. 命名ルール

### B1. テンプレート

```
{vendor}_{issue_date}_{amount_comma}_{project}.pdf
```

| フィールド | 説明 | 例 |
|-----------|------|-----|
| `{vendor}` | 業者名（正式名称または略称） | `杉崎リース工業株式会社`, `キョーワ(株)` |
| `{issue_date}` | 請求日（YYYYMMDD） | `20260125` |
| `{amount_comma}` | 金額（カンマ区切り） | `91,450`, `6,660,000` |
| `{project}` | 工事名・案件名 | `株式モビテック/シン・テクニカルセンター新築工事` |

- **根拠**: scope_freeze (Section 2 "命名形式"), config (`file_name_template`), 動画分析 (step_89 リネームダイアログで確認)
- **ステータス**: 確定（scope_freeze で凍結済み）

> **注意**: 3つのソースで不一致があった（下記参照）。scope_freeze の凍結値を正とする。
>
> | ソース | テンプレート | 差異 |
> |--------|------------|------|
> | scope_freeze / config | `{vendor}_{issue_date}_{amount_comma}_{project}.pdf` | **正** |
> | シナリオ README | `{vendor_short}__{issue_date}__{amount_comma}円.pdf` | ダブルアンダースコア、「円」suffix、工事名なし |
> | 2/13 ヒアリング | `{工事名称}_{業者名}.pdf` | 順序逆、日付/金額なし |
> | 動画実態 | `{業者名}_{YYYYMMDD}_{金額}_{工事名}.pdf` | scope_freeze と一致 |

### B2. 各フィールドの抽出優先順位

**全体方針**: filename-first（添付ファイル名を最優先）→ OCR フォールバック

```
1. 添付ファイル名からパース（filename-first）
   - 成功率: 66.4%（149件中99件）
   - Pattern A: {業者名}_{日付}_{金額}_{工事名}.pdf → 44.3%（完全形）
   - Pattern B: {業者名}_{日付}_{金額}.pdf → 22.1%（工事名なし）

2. PDF テキスト抽出（PyMuPDF / pypdf）
   - 成功率: 約30%（スキャンPDFは失敗）

3. OCR フォールバック（Tesseract → YomiToku）
   - 速度: Tesseract 5-10秒, YomiToku 60-100秒/ページ
   - 品質課題: 業者名の誤認識あり（トーケン→トータン 等）
```

| フィールド | 優先順位 |
|-----------|---------|
| `vendor` | ファイル名 → PDF本文 → OCR |
| `issue_date` | ファイル名 → PDF本文 → OCR |
| `amount_comma` | ファイル名 → PDF本文 → OCR |
| `project` | ファイル名（Pattern Aのみ: 44.3%） → PDF本文 → OCR → 空欄 |

- **根拠**: 設計ログ (2026-02-18 [G8] 全量テスト, [G3] パイロット結果), config (`filename_first: true`)
- **ステータス**: 確定

### B3. `{project}` フィールドの抽出元

| 優先順 | ソース | 成功率 | 備考 |
|--------|--------|--------|------|
| 1 | 添付ファイル名の第4セグメント | 44.3% | `_` 区切りの4番目 |
| 2 | PDF本文からの工事名抽出 | 低い | テキスト抽出できる場合のみ |
| 3 | OCR結果からの工事名抽出 | 低い | 速度・精度の問題あり |
| 4 | 空欄（省略） | — | 取得不可時はフィールドを空にする |

- `{project}` が取得不可でも命名失敗にはしない（vendor/date/amount があれば有効なリネーム）
- **根拠**: 設計ログ (G8全量テスト), パイロット結果
- **ステータス**: 確定

### B4. 命名失敗時のルール

| 条件 | 動作 |
|------|------|
| vendor/date/amount のいずれかが取得不可 | ファイル名先頭に `要確認_` を付与 |
| 取得できたフィールドは埋める | 例: `要確認_不明_20260125_91,450_.pdf` |
| 後工程（振り分け・印刷）は止めない | `要確認_` 付きファイルも通常通り保存・印刷する |

- **根拠**: scope_freeze (Section 2 "命名失敗"), MVP成功条件 #3
- **ステータス**: 確定

### B5. 既存ファイルと重複した場合の扱い

| 条件 | 動作 |
|------|------|
| 同名ファイルが保存先に存在 | `__dup` サフィックスを付与して別名保存 |
| 冪等性の課題 | 2回実行で `__dup` が生成される（H3指摘） |

- **根拠**: 設計ログ (H3指摘), パイロット結果
- **ステータス**: 要確認
  - **確認事項**: SHA-256重複チェックの実装要否（冪等性テスト G7 未実施）
  - **確認先**: T1 (実装チーム)

## C. フォルダ振り分けルール

### C1. ルーティング方式（MVP）

MVP では **キーワードマッチ方式** を採用（LCS/3gram は MVP 対象外）。

### C2. 振り分け先定義

| サブフォルダ | キーワード | マッチ条件 |
|-------------|-----------|-----------|
| `モビテック` | `["モビテック", "シン・テクニカルセンター", "3329"]` | いずれか1つを含む |
| `中村` | `["中村", "中島三丁目", "3285"]` | いずれか1つを含む |

- **根拠**: config (`routing.rules`), scope_freeze (Section 3 "LCS/3gram は keyword routing で代替"), 動画分析 (step_141-147 で `ビデック`/`中村`/`その他` のサブフォルダ確認)
- **ステータス**: 確定

### C3. ルーティング検索対象フィールド（優先順）

| 優先順 | 検索対象 | 備考 |
|--------|---------|------|
| 1 | `{project}` (工事名) | ファイル名/OCRから取得した工事名 |
| 2 | `attachment_name` (添付ファイル名) | 元の添付ファイル名全体 |
| 3 | `subject` (メール件名) | Outlook メールの件名 |
| 4 | `vendor` (業者名) | 抽出した業者名 |

- キーワードを上記フィールドの順に検索し、最初にマッチした時点で振り分け先を決定
- どのフィールドにもマッチしなければ `その他`

- **根拠**: 設計ログ (3文字一致ロジックの `source_priority: ["filename", "pdf_text"]`), config
- **ステータス**: 要確認
  - **確認事項**: 検索フィールドの順序と対象（実装時にT1と合意が必要）
  - **確認先**: T1 (実装チーム)

## D. FlagStatus 運用ルール

### D1. 状態遷移図

```
NoFlag (0)                    ← 初期状態（未処理）
  │
  ▼  [RPA処理成功]
Flagged (2)                   ← RPA処理済み・レビュー待ち
  │
  ▼  [月次クローズ（手動）]
Complete (1)                  ← 月次完了
```

### D2. 各状態の定義

| FlagStatus | 値 | 意味 | セットする主体 |
|------------|-----|------|--------------|
| NoFlag | 0 | 未処理。RPA の処理対象 | メール受信時のデフォルト |
| Flagged | 2 | 処理済み・レビュー待ち | **RPA** が自動セット (`mark_flag_on_success: true`) |
| Complete | 1 | 月次完了（PDF保存・振分け・印刷・確認 全て完了） | **人間**（藤田さん）が手動セット |

### D3. RPA の FlagStatus 操作

| イベント | 操作 |
|---------|------|
| 処理成功（全添付PDF保存完了） | `FlagStatus = 2` に変更 |
| 処理失敗（一部でもエラー） | `FlagStatus` 変更しない（0のまま） |
| FlagStatus 読取失敗（COM例外） | スキップ（H2対策: 安全フォールバック） |

### D4. 月次クローズ手順

| 手順 | 担当 | 操作 |
|------|------|------|
| 1 | 藤田さん | RPA処理結果（`Flagged=2` のメール群）をOutlookで目視レビュー |
| 2 | 藤田さん | 保存先フォルダのPDFが正しいことを確認 |
| 3 | 藤田さん | 確認完了メールの FlagStatus を `Complete (1)` に手動変更 |
| 4 | 藤田さん | 必要に応じてメールを月次アーカイブフォルダ（`202601` 等）に移動 |

- **根拠**: 設計ログ (2026-02-17 [CONFIG], [DATA] 教師データ分析), config
- **ステータス**: 要確認
  - **確認事項 (G4)**: 状態遷移と月次クローズ手順について藤田さんの合意が必要
  - **確認先**: 藤田さん
  - **リスク (M1)**: `Flagged(2)` が「手動作業中」と「RPA処理済」の両方の意味を持つ二重化問題

## E. 例外ルール

### E1. パスワード付きPDF（HENNGE）

| 項目 | 値 |
|------|-----|
| MVP動作 | **スキップ**（処理しない） |
| 設定値 | `enable_pdf_decryption: false` |
| 件名フィルタ | `パスワード`, `HENNGE` は件名フィルタに含まれる（候補には入る） |
| 実際の動作 | 候補に入るがPDF復号しないため、暗号化PDFはそのまま保存 |

- **根拠**: scope_freeze (Section 3 "HENNGE復号"), config
- **ステータス**: 確定（MVP対象外）

### E2. ZIP ファイル

| 項目 | 値 |
|------|-----|
| MVP動作 | **スキップ**（展開しない） |
| 設定値 | `enable_zip_processing: false` |
| 全量テストでの検出 | 9件のZIP添付あり |

- **根拠**: scope_freeze (Section 3), 設計ログ (G8全量テスト: ZIP 9件)
- **ステータス**: 確定（MVP対象外）

### E3. ✖/× マーク付きファイル

| 項目 | 値 |
|------|-----|
| ✖ の意味 | 支払対象外・無効（修正前版、発注キャンセル、二重請求除外） |
| MVP動作 | **スキップ + 警告ログ記録** |
| 実装方式 | ファイル名に `✖` または `×` を含む場合は保存しない |

**実データでの確認例**:
```
3278_ASADA/
  ├── ✖太田商事20250525_43,200,000【ARTCLINICASADA】.pdf   ← 無効版
  └── 太田商事20250525_4,278,600【ARTCLINICASADA】.pdf     ← 有効版
```

- **根拠**: 設計ログ (2026-02-19 [DISCOVERY] Codex分析), scope_freeze (Section 3)
- **ステータス**: 確定（スキップ+ログ）
  - **リスク (Q3)**: RPAは添付ファイル名中の✖を検出するが、PDF本文中の✖マークは検出しない

### E4. 複数添付メール

| 項目 | 値 |
|------|-----|
| 動作 | 各添付ファイルを**個別に**処理（1メール = N個のPDF → N回のリネーム・振り分け） |
| FlagStatus更新 | 全添付PDFの処理が成功した場合のみ `Flagged(2)` に変更 |
| 部分成功時 | FlagStatus は変更しない（0のまま）。次回実行時に再処理 |

- **根拠**: 設計ログ (H3指摘 "部分成功時の冪等性"), パイロット結果
- **ステータス**: 確定
  - **リスク (H3)**: 部分成功→再実行で成功済みファイルが `__dup` として重複保存される

### E5. 鑑+明細セット（PDF結合）

| 項目 | 値 |
|------|-----|
| MVP動作 | **結合しない**（個別ファイルとして保存） |
| 設定値 | `merge.enabled: false` |
| 動画での実態 | 藤田さんは iLovePDF (Web) で手動結合している |

- **根拠**: scope_freeze (Section 3 "PDF結合"), config, 動画分析 (step_82-119)
- **ステータス**: 確定（MVP対象外）
  - **将来課題 (Q7)**: pypdf等でのローカルPDF結合の実装判断

### E6. 非準拠ファイル名（Pattern D）

| 項目 | 値 |
|------|-----|

| 内訳 | 日付フォーマット違い(30%), 区切り文字違い(25%), 数値フォーマット違い(20%), スキャナー自動命名等(25%) |

- **ステータス**: 確定
